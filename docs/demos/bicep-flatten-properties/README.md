# Testing PR #12132 — flat `properties` aliases for manifest-generated Bicep types

[PR #12132](https://github.com/radius-project/radius/pull/12132) makes the Go
manifest-driven Bicep type generator (`bicep-tools/pkg/converter`) hoist the
children of a resource's `properties` object onto the resource body as ReadOnly
**flat aliases**. This brings the Go pipeline to parity with the TypeScript
autorest pipeline, so Bicep authors of any manifest-generated namespace
(`Radius.Compute`, `Radius.Data`, `Radius.Security`, and any user type published
via `rad bicep publish-extension`) can read flat references such as
`widget.message` instead of `widget.properties.message`.

This folder is a self-contained, end-to-end test of that feature using a custom
resource type and a real Bicep deployment.

## Files

| File                  | Purpose                                                                                                                            |
|-----------------------|------------------------------------------------------------------------------------------------------------------------------------|
| `widgets.yaml`        | A custom resource type manifest (`MyCompany.Test/widgets`) with `message`, `count`, and `enabled` properties.                      |
| `deploy-flat.bicep`   | A deployment that reads the new flat aliases (`widget.message`, `widget.count`, `widget.enabled`). This is the feature under test. |
| `deploy-nested.bicep` | A deployment that reads the nested envelope (`widget.properties.message`) to confirm backward compatibility.                       |
| `bicepconfig.json`    | Registers the locally built extension `mycompanytest` (the `.tgz` produced by the steps below).                                    |

## Prerequisites

- Go (see `go.mod` for the pinned version) to build the generator.
- The `bicep` CLI (the test was verified with Bicep CLI 0.42.1).

## Steps

Run from this directory.

1. Build the `manifest-to-bicep` generator from the PR branch:

   ```bash
   go build -o /tmp/manifest-to-bicep ./bicep-tools/cmd/manifest-to-bicep
   ```

2. Generate the Bicep extension for the custom resource type:

   ```bash
   /tmp/manifest-to-bicep generate widgets.yaml ./out
   ```

3. Package the extension into the local `.tgz` referenced by `bicepconfig.json`:

   ```bash
   bicep publish-extension ./out/index.json --target ./mycompanytest.tgz --force
   ```

4. Compile the deployment that uses the flat aliases:

   ```bash
   bicep build deploy-flat.bicep --outfile ./deploy-flat.json
   ```

5. Compile the deployment that uses the nested envelope form:

   ```bash
   bicep build deploy-nested.bicep --outfile ./deploy-nested.json
   ```

## Verification

- **Generated types expose flat aliases.** In `out/types.json`, the resource
  body object lists `message`, `count`, and `enabled` alongside the envelope
  keys (`name`, `location`, `properties`, `apiVersion`, `type`, `id`). Each
  hoisted alias has `flags: 2` (ReadOnly only).

- **Flat deployment compiles (the feature).** `bicep build deploy-flat.bicep`
  succeeds. The compiled ARM JSON resolves the aliases to the underlying
  properties, for example:

  ```json
  "widgetMessage": { "type": "string", "value": "[reference('widget').message]" }
  ```

- **Nested deployment still compiles (backward compatible).**
  `bicep build deploy-nested.bicep` succeeds, confirming the original
  `properties` envelope is preserved.

- **Negative control (without the PR).** When the extension is generated by the
  `main` build of `manifest-to-bicep`, the flat aliases are absent and
  `deploy-flat.bicep` fails to compile with `BCP053`:

  ```text
  Error BCP053: The type "MyCompany.Test/widgets@2025-01-01-preview" does not
  contain property "message". Available properties include "apiVersion", "id",
  "location", "name", "properties", "type".
  ```

  This confirms the flat aliases are introduced by PR #12132 and are what allow
  the flat deployment to compile.
