# GitHub Workspace Type

This document describes the `github` workspace kind for Radius, which is designed for use inside
GitHub Actions workflows. It automates the lifecycle of a local Kubernetes cluster (via
[k3d](https://k3d.io)) with Radius installed and a PostgreSQL backend, backed by a git orphan
branch for state persistence across workflow runs.

## Overview

A workspace of kind `github`:

1. Creates a k3d Kubernetes cluster on `rad init --kind github`.
2. Installs Radius with PostgreSQL as the state backend (`database.enabled=true`).
3. Restores any previously committed database state from the `radius-state` orphan branch.
4. On `rad shutdown`, backs up the PostgreSQL state to the `.radius/state/` directory and commits it
   to the `radius-state` branch.
5. Optionally tears down the k3d cluster (`rad shutdown --cleanup`).

## Prerequisites

- [k3d](https://k3d.io) v5+ installed and on `$PATH`.
- `kubectl` installed and on `$PATH`.
- A git repository (required for state management via orphan branch).

## Quick Start

### Initialize

```bash
rad init --kind github
```

This will:

1. Ensure `k3d` is available.
2. Create a k3d cluster named `radius-github` (idempotent — reuses existing cluster).
3. Install Radius with PostgreSQL using Helm (`database.enabled=true`).
4. Wait for PostgreSQL to be ready.
5. Restore database state from the `radius-state` orphan branch (if it exists).
6. Create a workspace named `default` with `kind: github`.

You can customize the state directory:

```bash
rad init --kind github --state-dir /path/to/custom/state
```

### Shut down

```bash
rad shutdown
```

This will:

1. Dump each PostgreSQL database to a plain SQL file in the state directory.
2. Commit the state directory contents to the `radius-state` orphan branch.
3. Print instructions to push the state to the remote.

To also delete the k3d cluster:

```bash
rad shutdown --cleanup
```

> **Note:** `rad shutdown` does **not** push to the remote. Your GitHub Actions workflow must run
> `git push origin radius-state` after `rad shutdown` to persist state between runs.

## State Directory

The default state directory is `.radius/state/` relative to the current working directory.
The directory contains one SQL file per database:

```
.radius/state/
  ucp.sql           # UCP (Universal Control Plane) database backup
  applications_rp.sql  # Applications RP database backup
```

These are plain SQL files generated by `pg_dump --format=plain --clean --if-exists`.

## Orphan Branch: `radius-state`

State is committed to a git orphan branch named `radius-state`. This branch:

- Is **never merged** into `main` or any feature branch.
- Contains only the `.radius/state/` directory contents.
- Uses commit messages of the form `radius state backup 2026-03-18T12:00:00Z`.

To inspect the state branch:

```bash
git log radius-state --oneline
git show radius-state:.radius/state/ucp.sql | head -20
```

To manually restore state from the orphan branch:

```bash
git checkout radius-state -- .radius/state/
```

## State Persistence Across Runs

The `gitstate` package commits SQL backup files to a local `radius-state` orphan branch on the
runner. Within a **single workflow job** this is enough — the local git repo persists between
steps, so `rad shutdown` commits and `rad init` restores without any push or upload.

To persist state **across workflow runs** (the normal production case for a demo repo), upload
the state directory as a GitHub Actions artifact after shutdown and download it before init:

```yaml
jobs:
  run-with-radius:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      # Download state from a previous run (no-op on first run).
      - name: Download Radius state
        uses: actions/download-artifact@v4
        continue-on-error: true   # first run: no artifact exists yet
        with:
          name: radius-state
          path: .radius/state/

      - name: Initialize Radius
        run: rad init --kind github

      - name: Run tests / deploy app
        run: # ...

      - name: Shut down Radius
        run: rad shutdown --cleanup

      # Upload state so the next run can restore it.
      - name: Upload Radius state
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: radius-state
          path: .radius/state/
          retention-days: 90
          overwrite: true
```

> **Why artifacts instead of git push?**
> - No write access to the repository required.
> - SQL backup files don't pollute the git history of the demo repo.
> - Artifacts are scoped to the workflow and can be managed independently of code.
> - The `radius-state` orphan-branch mechanism in `gitstate` is still used for the **local**
>   commit within a run, but the artifact is the cross-run persistence layer.

> **Artifact retention:** GitHub retains artifacts for 90 days by default (configurable). If
> your demo workflow runs less frequently than that, increase `retention-days` accordingly.

## GitHub Actions Example

A minimal workflow for a demo repo using artifact-based state persistence:

```yaml
name: Demo with Radius

on:
  push:
    branches: [main]

jobs:
  demo:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install k3d
        run: curl -s https://raw.githubusercontent.com/k3d-io/k3d/main/install.sh | bash

      - name: Install rad CLI
        run: # ... install rad from release

      - name: Download prior Radius state (if any)
        uses: actions/download-artifact@v4
        continue-on-error: true
        with:
          name: radius-state
          path: .radius/state/

      - name: Initialize Radius
        run: rad init --kind github

      - name: Deploy / test application
        run: # ...

      - name: Shut down Radius
        run: rad shutdown --cleanup
        if: always()

      - name: Save Radius state
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: radius-state
          path: .radius/state/
          retention-days: 90
          overwrite: true
```

## Integration Test Workflow

A dedicated integration test workflow is provided at
[.github/workflows/functional-test-github-workspace.yaml](.github/workflows/functional-test-github-workspace.yaml).
It exercises the full lifecycle on a GitHub Actions runner **within a single job**:

1. **Phase 1 — Init (first run):** `rad init --kind github` creates the k3d cluster and installs
   Radius with PostgreSQL. No prior state exists on this run.
2. **Phase 2 — Deploy demo content:** Currently a placeholder (`echo "No demo application yet"`).
   See §[Demo Application](#demo-application) below for how to extend this phase.
3. **Phase 3 — Shutdown:** `rad shutdown --cleanup` backs up PostgreSQL to `.radius/state/` and
   commits it to the local `radius-state` orphan branch. The k3d cluster is deleted. Because
   this is a single job, the local git repo persists on-disk between steps — **no push to any
   remote is needed**.
4. **Phase 4 — Restore:** `rad init --kind github` runs again on a fresh cluster and restores
   state from the local `radius-state` branch. `make workflow-github-workspace-verify-restore`
   asserts the SQL files exist on that branch.
5. **Phase 5 — Cleanup:** Deletes the k3d cluster unconditionally (`always()`) and uploads logs
   on failure.

The workflow triggers on:
- Pull requests touching the workspace / k3d / pgbackup / gitstate / Helm chart code paths.
- `workflow_dispatch` for manual runs during development.

### Make Targets

The workflow delegates to Make targets for local testability:

| Target | Purpose |
|--------|---------|
| `make workflow-github-workspace-init` | `rad init --kind github --state-dir .radius/state` |
| `make workflow-github-workspace-shutdown` | `rad shutdown --cleanup` |
| `make workflow-github-workspace-verify-restore` | Asserts SQL backup files exist on the state branch |

You can run these locally against a k3d cluster to debug the workflow without pushing.

## Demo Application

The integration test workflow has a placeholder **Phase 2** step where a demo application
would be deployed and verified before shutdown. To wire up a real demo app:

1. Create (or point to) a repository containing a Radius application — a Bicep file with
   `Applications.Core/containers` and whatever portable resources it needs.
2. In the integration workflow's **Phase 2** step, replace the placeholder with:
   ```yaml
   - name: Deploy demo application
     run: |
       rad deploy path/to/app.bicep
       rad resource list Applications.Core/containers
   ```
3. In **Phase 4 (Verify restore)**, add assertions that the application resources are still
   present after the state restore, e.g.:
   ```yaml
   - name: Verify application resources restored
     run: |
       rad resource show Applications.Core/containers mycontainer
   ```
4. Update `make workflow-github-workspace-verify-restore` in `build/workflow.mk` to cover
   application-level assertions in addition to the SQL file check.

The demo app does not need to be in this repository — it can be a separate repo checked out
in a workflow step. The key requirement is that it uses only portable resources (Redis, SQL, etc.)
backed by Radius recipes so that the state lives entirely in PostgreSQL and round-trips through
the backup/restore cycle.

## Architecture

```
rad init --kind github
    │
    ├── k3d.EnsureInstalled()         Check k3d is on PATH
    ├── k3d.CreateCluster()           k3d cluster create radius-github --wait --timeout 120s
    ├── helm.InstallRadius()          Helm install with database.enabled=true
    ├── pgbackup.WaitForReady()       kubectl wait --for=condition=ready pod -l app.k8s.io/name=database
    ├── gitstate.RestoreState()       git checkout radius-state -- .radius/state/  (if branch exists)
    └── pgbackup.Restore()            kubectl exec psql < .radius/state/{db}.sql  (if backup exists)

rad shutdown
    │
    ├── pgbackup.Backup()             kubectl exec pg_dump > .radius/state/{db}.sql
    ├── gitstate.CommitState()        git add .radius/state/ && git commit -m "radius state backup ..."
    └── k3d.DeleteCluster()           k3d cluster delete radius-github  (only with --cleanup)
```

## Package Reference

| Package | Path | Purpose |
|---------|------|---------|
| `k3d` | `pkg/cli/k3d` | k3d cluster lifecycle management |
| `pgbackup` | `pkg/cli/pgbackup` | PostgreSQL backup/restore via `kubectl exec` |
| `gitstate` | `pkg/cli/gitstate` | Orphan branch state management via `git` |

## PostgreSQL Databases

Two databases are backed up and restored:

| Database | Used By |
|----------|---------|
| `ucp` | UCP (Universal Control Plane) and Dynamic RP |
| `applications_rp` | Applications RP |

## Manual Restore

If you need to manually restore PostgreSQL state without running `rad init`:

```bash
# Restore state files from git
git checkout radius-state -- .radius/state/

# Restore into a running Radius cluster
POD=$(kubectl -n radius-system get pods -l app.kubernetes.io/name=database \
  -o jsonpath='{.items[0].metadata.name}')

kubectl -n radius-system exec -i "$POD" -- psql -U radius -d ucp \
  < .radius/state/ucp.sql

kubectl -n radius-system exec -i "$POD" -- psql -U radius -d applications_rp \
  < .radius/state/applications_rp.sql
```

## Helm Chart PostgreSQL Support

The `database.enabled=true` Helm value enables full PostgreSQL support:

- Deploys a PostgreSQL StatefulSet (`mirror/postgres:16-alpine`).
- Creates `ucp` and `applications_rp` databases on first startup via
  `/docker-entrypoint-initdb.d/` scripts.
- Injects `POSTGRES_PASSWORD` from a Kubernetes Secret into all RP pods.
- Configures all RPs to use PostgreSQL instead of the default Kubernetes API server store.

This PostgreSQL support is not exclusive to the `github` workspace kind — any workspace can use
`database.enabled=true` when installing Radius via Helm.
