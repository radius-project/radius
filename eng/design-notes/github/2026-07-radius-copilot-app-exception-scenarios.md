# Feature Specification: Radius Copilot App Integration Exception Scenarios

- **Author**: Zach Casper (@zachcasper)

## Summary

This document describes the error and exception scenarios for the Radius integration in the GitHub Copilot app, the user experience when each occurs, and how the user recovers or corrects the problem. It is scoped to the MVP (the initial public preview) of the integration.

It is organized around the parts of the end-to-end user flow already defined in the companion document, [Radius Integration in GitHub Copilot App](radius-copilot-app-integration.md). For each part, this document enumerates the possible errors and exceptions, what the user sees, and the path back to a good state.

The integration has three components, defined in the terms below: the **Radius skill** (in the Copilot chat), the **Radius side panel** (in the sidebar), and **Repo Radius** (which runs each operation as a GitHub Actions workflow). Where a failure originates in Repo Radius, this document references the Repo Radius outcome the skill or side panel surfaces rather than redefining it.

The MVP does not support repositories without a containerized workload, preview deployments via the planned application graph view, or deployments to targets other than AWS or Azure. Where a user attempts one of these, Radius tells them clearly that the configuration is unsupported rather than letting them proceed into an ambiguous failure.

## Definition of terms

- **Canvas**: The GitHub Copilot app's extension framework, which renders extensions in the sidebar. Canvas is a Copilot capability, not a Radius component.
- **Radius Canvas extension**: The Radius extension, built on the Canvas framework, that powers the Radius side panel.
- **Radius skill**: The Copilot skill that runs in the Copilot chat and drives the Radius flow. Invoked from a natural-language prompt or explicitly with the `/radius` command.
- **Radius plugin**: The GitHub Copilot app plugin that packages and versions the Radius skill and the Radius Canvas extension as one unit. Installed and updated from the Copilot app Plugins settings page.
- **Radius side panel**: The GUI for Radius, built using the Radius Canvas extension and shown in the Copilot app sidebar. It displays the application graph, forms, and operation status, and is the surface the user opens, views, and interacts with.
- **Repo Radius**: The component that runs each Radius operation as a GitHub Actions workflow and returns results as artifacts (for example, `run-rad-commands-result`, `verify-cloud-auth-result`). Defined in the [Repo Radius feature specification](https://github.com/radius-project/radius/pull/12078).
- **Assembly analysis**: The analysis the Radius skill runs on the repository to identify the application and its resources and produce the application definition.
- **Application definition**: The `app.bicep` file that describes the application, written into the `.radius` directory.
- **Application graph**: The graph of the application's resources and their relationships, shown in one of three views:
  - **Modeled application graph view**: Generated by compiling `app.bicep`; shows the application as defined, before deployment.
  - **Deployed application graph view**: Shows the resources actually deployed to the target environment.
  - **Planned application graph view**: A preview (what-if) of changes before deploying. Deferred for the MVP.

## Exceptions handling principles

These principles apply to all exception scenarios:

- **Every exception is informative and helpful**: it tells the user what went wrong and how to fix it. Concretely, every error carries three things: a plain-language description of the problem (what happened), remediation guidance (the change the user makes to correct the cause), and a recovery action (the button, command, or follow-up request the user uses to try again from where they are).
- **Prefer pre-flight checks over in-progress failures**. A pre-flight failure (a missing setting, a disallowed command, an unsupported configuration) is corrected and retried with no cleanup, while an in-progress failure during a running deployment or deletion may leave the environment partially changed and require converging or a delete-and-redeploy. Radius should validate as much as possible up front.
- **Surface each error where the user can act on it**. An error appears at the point of action, not in a distant log the user has to hunt for. A failure that originates in Repo Radius is surfaced to the user by the Radius skill or side panel with its specific outcome and remediation guidance, rather than left as a raw or generic failure.
- **Present each exception in one of two channels, each with its own voice**. In the chat, the skill answers in the first person, as a direct response to what the user asked (for example, "I can't find a repository attached to this session. Add a repository and try again"), and never refers to Radius by name. In the Radius side panel, exceptions are GUI states shown where the user is working: inline form validation, a graph node in a failed state exposing its error, or a render error. A single failure may be acknowledged in chat and detailed in the side panel.
- **Always give the user a path back to a good state**. No failure is a dead end: every one resolves through one of three recovery patterns. Correct-and-retry (fix a setting or the application definition, then run again), re-verify (re-run cloud authentication after fixing trust or permissions), and delete-and-redeploy (tear the application down and provision it fresh when converging cannot recover a good state).

## Part 0: Discover, install, and update the Radius plugin

Discovering, installing, updating, and removing the plugin is Copilot app functionality, not Radius. The Copilot app handles the failures below and their error reporting; Radius implements no recovery for them. They are described here only so the reader knows where they originate and does not mistake them for Radius exceptions.

### Exception 0.1: Plugin install fails

The user has found the Radius plugin on the Plugins settings page and chosen to add it, but the install does not run to completion, usually because of a transient network or download error. The plugin remains listed but is not installed.

- **What the user sees:** The install does not complete and the settings page reports the failure.
- **Recovery:** Handled by the Copilot app. The user retries the install; a transient error usually clears on retry.

### Exception 0.2: Update fails

The Radius plugin is already installed and working, and the user starts an update to a newer version, but the update does not run to completion. The previously installed version is still in place.

- **What the user sees:** The update does not complete and the settings page reports the failure.
- **Recovery:** Handled by the Copilot app. The user retries the update later; the existing version keeps working in the meantime.

## Part 1: Clone the repo, invoke the Radius skill, and open the Radius side panel

Before any Assembly analysis can begin, the repository must be brought into the session and checked out as a worktree, the Radius skill must fire and enable the repository for Radius, and the Radius side panel must come up in the sidebar. Bringing the repository in is a Copilot app prerequisite; once it is available, the first thing the skill does is enable the repository by committing the Repo Radius workflow files into `.github/workflows`, because Repo Radius runs every operation as a GitHub Actions workflow that must be committed before it can run. The exceptions here are the repository not being available (1.1), the skill not being invoked (1.2), enablement failing for lack of write permission (1.3), and the Radius side panel not coming up (1.4). All are pre-flight, so nothing has been analyzed or deployed yet, and the user recovers by correcting the cause and re-invoking the skill with no cleanup.

### Exception 1.1: Repository not available in the session

The Radius skill needs the repository checked out before it can do anything. Bringing the repository in and checking it out as a worktree is Copilot app functionality, so the ways it can fail are a prerequisite rather than Radius behavior: no repository is attached, a clone or checkout fails (a transient error, or a private repository the user cannot access), or a clone times out on a very large repository. If the repository is missing, the skill says so; otherwise the Copilot app reports the failure. Either way, Radius cannot proceed until the repository is checked out.

- **What the user sees:** If no repository is attached, Copilot responds in chat: "I don't see a repository attached to this session yet. Add one and I'll take a look." If a repository is attached but cannot be cloned or checked out, the Copilot app reports that failure.
- **Recovery:** The user makes sure a repository is attached and successfully checked out: add one if none is attached, confirm access if it is private, and retry if a transient error or a very large repository caused the clone to fail. Then the user re-invokes the Radius skill.

### Exception 1.2: Skill not invoked from a natural-language prompt

The session is ready with a repository attached, and the user asks Copilot something that should invoke the Radius skill, such as asking to see the application graph. Copilot answers generally and never invokes the Radius skill, so none of the Radius flow runs and the Radius side panel never appears.

- **What the user sees:** A generic Copilot chat response with no Radius side panel and no application graph.
- **Recovery:** The user invokes the skill explicitly with the `/radius` command (named after the plugin the user enabled), which is the deterministic entry point.

### Exception 1.3: Repository cannot be enabled for Radius

Enabling the repository is the first write to the repository. If the branch is protected or the user lacks write permission, committing the workflow files fails and Radius cannot proceed. This only affects a repository that has not been enabled yet: one enabled on an earlier session already has the workflow files (and its `app.bicep` application definition), so a read-only user working with an already-enabled repository never reaches this exception.

- **What the user sees:** Copilot responds in chat: "To set this repository up I need to add a couple of workflow files, but I don't have write access to <repository> in this session. If you fork the repository or get write access and add it here, I can pick up where we left off."
- **Recovery:** The user forks the repository or gets write access, adds it to the session, and re-invokes the Radius skill.

### Exception 1.4: Radius side panel does not open in the sidebar

The Radius skill runs and responds in chat, but the Radius side panel never appears in the sidebar. The plugin installs the skill and the Radius Canvas extension together, so the extension is present but failed to load (a crash on startup, a user modification, or an incompatible or corrupted build), and cannot build the side panel. The skill must detect the unavailable extension and report it clearly rather than appearing broken.

- **What the user sees:** The sidebar shows no Radius side panel, or the panel shows a load error, and Copilot responds in chat: "The Radius plugin didn't load completely, so I can't show its view in the sidebar. Reinstalling usually fixes this: remove the Radius plugin from the Plugins settings page, add it again, then restart the Copilot app."
- **Recovery:** The user restarts the Copilot app to give the extension another chance to load. If it still does not appear, they re-enable it (if disabled) or reinstall the Radius plugin for a clean build of both the skill and the extension.

## Part 2: Create the resource types and application definition

Part 2 is authoring: Copilot runs the Assembly analysis on the repository and writes `app.bicep`, `custom-types.yaml`, and `bicepconfig.json` into a new `.radius` directory. The most important pre-flight check is the presence of a Dockerfile, because a repository without a containerized workload is out of scope for the MVP.

### Exception 2.1: No Dockerfile present

The user asks Copilot to create an application definition, but the repository does not contain a Dockerfile. Because the MVP only supports containerized workloads, Copilot cannot create an application definition for it.

- **What the user sees:** Copilot responds in chat: "I could not find a Dockerfile in this repository. I can only create application definitions for containerized applications. Add a Dockerfile first, then I can create an application definition."
- **Recovery:** The user adds a Dockerfile for their workload and asks Copilot to analyze the repository again.

### Exception 2.2: No recognizable application

As Copilot analyzes the repository, it cannot identify an application or its application resources. Nothing is written.

- **What the user sees:** Copilot responds in chat: "I looked through the repository but could not identify an application or application resources. Which directory contains your application source code and Dockerfile?"
- **Recovery:** The user responds with a directory name in the repository containing identifiable application resources, then asks Copilot to analyze the repository again.

### Exception 2.3: An application definition already exists

The repository already contains a `.radius` directory with an `app.bicep` application definition from a previous Assembly analysis, and the user asks Copilot to analyze the repository again. Rather than silently overwriting the existing `app.bicep`, Copilot stops and asks how to proceed.

- **What the user sees:** Copilot responds in chat: "This repository already has an application definition in the `.radius` directory. Would you like me to keep it, or analyze the repository again and overwrite `app.bicep`?"
- **Recovery:** The user chooses to keep the existing definition, or to regenerate it, which overwrites the existing `app.bicep`.

### Exception 2.4: Assembly analysis fails

The user starts the Assembly analysis on a repository, but it does not successfully complete, for example because of a transient model or tooling error. Nothing is written on a failed Assembly analysis.

- **What the user sees:** Copilot responds in chat: "I wasn't able to finish analyzing the repository this time. Want me to try again?"
- **Recovery:** The user retries; the Assembly analysis is re-runnable and safe to repeat because nothing is written until it succeeds.

### Exception 2.5: Cannot write the `.radius` files

Copilot finishes analyzing the repository and has the generated files ready, but it cannot commit them to the working branch, for example because the branch is protected or the user lacks write permission. No `.radius` directory is created.

- **What the user sees:** Copilot responds in chat: "I generated the `.radius` files but couldn't write them to <branch>. Switch to a branch I can write to and I'll save them."
- **Recovery:** The user switches to a branch Copilot can write to, creating a new branch or using a fork if the target branch is protected, then asks Copilot to save the `.radius` files again so the `.radius` directory is created on that branch.

### Exception 2.6: Generated `app.bicep` is invalid

After the Assembly analysis writes the `.radius` directory, Copilot compiles `app.bicep` to generate the modeled application graph view. If the generated `app.bicep` does not compile, the view cannot be generated, and Copilot reports the compile error instead of showing the graph.

- **What the user sees:** Copilot responds in chat: "I built the application definition, but the `app.bicep` didn't compile: <compile error>. I can try to fix it, or you can edit it directly."
- **Resulting state:** A `.radius` directory with an invalid definition that exists but cannot be rendered or deployed until corrected.
- **Recovery:** The user asks the skill to correct the definition, or edits `app.bicep` directly, then Copilot regenerates the modeled application graph view.

## Part 3: View the application graph

The Radius side panel displays the modeled application graph view that the Assembly analysis generated and stored in the `.radius` directory. Because this is the pre-generated modeled view rather than a live query, failures here are about loading or rendering that view, not about deployed infrastructure.

### Exception 3.1: Modeled application graph view missing or unreadable

The Assembly analysis completed successfully, so `app.bicep` and the modeled application graph view were both written to the `.radius` directory. But when the user opens the Radius side panel, the modeled application graph view does not appear, because it is missing or cannot be read, so the side panel has nothing to display.

- **What the user sees:** Copilot responds in chat: "I cannot find the modeled application graph view to show. I can rebuild it from the application definition then show it."
- **Recovery:** Copilot automatically regenerates the modeled application graph view from the `app.bicep` file and renders the view in the Radius side panel.

### Exception 3.2: Empty application graph

The user opens the Radius side panel and the modeled application graph view loads, but it contains no resources, so the side panel renders an empty graph.

- **What the user sees:** The Radius side panel renders but shows no resources on the graph. In the center of the empty graph is a warning: "There are no application resources for this application" followed by a button labeled "Re-generate modeled application graph view" and "Re-create the application definition."
- **Recovery:** The user clicks the re-generate button and the Radius side panel re-generates the modeled application graph view from `app.bicep`, or clicks the re-create application definition button to run the Assembly analysis again.

### Exception 3.3: Radius side panel render error

The modeled application graph view is present and readable, but when the user opens the Radius side panel it fails to draw the graph because of a transient rendering error, leaving it blank or errored.

- **What the user sees:** The Radius side panel fails to draw the application graph and shows a render error.
- **Recovery:** The user reopens the Radius side panel by re-invoking the skill, which reloads the Radius Canvas extension.

## Part 4: Create a cloud environment (AWS or Azure)

Part 4 is the user creating an environment. From the Environments tab of the Radius side panel, the user clicks **Create New Environment**, selects AWS or Azure, provides an environment name, enters their credentials, then clicks **Verify Credentials**. This dispatches the `radius-verify-cloud-auth` workflow, which produces the `verify-cloud-auth-result` artifact. Several exceptions can surface along the way.

### Exception 4.1: Required environment property missing or invalid format

The user is guided through configuring OIDC, then enters the cloud account and credential information on the **Create Environment** form in the Radius side panel. For Azure, this includes the client ID, tenant ID, subscription ID, and resource group name. For AWS, this includes the account ID, IAM role name, and region. If one of these properties is not provided or is invalid, an exception occurs.

- **What the user sees:** If the user does not populate one of the fields on the **Create Environment** form before clicking the **Verify Credentials** button, a pre-flight form validation error tells the user the field is required. The form validation checks that each field is populated and in the correct format.
- **Recovery:** The user supplies the missing value or corrects the format and clicks **Verify Credentials**.

### Exception 4.2: GitHub Environment variables not set correctly

Once the user provides all required data in the proper format and clicks **Verify Credentials**, the `radius-verify-cloud-auth` workflow is dispatched. If the environment variables on the GitHub Environment were not created correctly by the Radius Canvas extension, the verify workflow fails and the `verify-cloud-auth-result` artifact includes a **Required setting missing** error.

- **What the user sees:** After clicking **Verify Credentials**, the user is shown a modal status box: "Verifying authentication to <cloud provider>. This may take a few moments." When the verify workflow fails, the Radius side panel reads the `verify-cloud-auth-result` artifact and shows: "One or more required environment properties are missing. Delete the GitHub Environment and recreate the environment in the Radius side panel."
- **Recovery:** The user deletes the GitHub Environment and recreates the environment in the Radius side panel. Because this exception is highly unlikely, manual recovery by the user is acceptable.

### Exception 4.3: Cloud provider does not trust GitHub

The user must manually create an OIDC trust relationship between the cloud provider and Copilot, following guidance from Radius. If the GitHub federated identity provider has not been added as a trusted identity provider, then after the user clicks **Verify Credentials**, the verify workflow returns a "Cloud provider does not trust GitHub" error in the `verify-cloud-auth-result` artifact.

- **What the user sees:** When the verify workflow fails, the Radius side panel reads the `verify-cloud-auth-result` artifact and shows: "The cloud provider rejected the GitHub OIDC token. Verify that OIDC is configured correctly, then click **Verify Credentials** again."
- **Recovery:** The user corrects the trust configuration (the AWS IAM role's trust policy, or the Azure Entra ID app registration's federated credential), then clicks **Verify Credentials** again.

### Exception 4.4: Identity lacks required permissions

After the user clicks **Verify Credentials**, the verify workflow checks that the identity (the AWS IAM role or the Azure identity) has sufficient permissions in the cloud provider to manage application resources. If any permissions are missing, the verify workflow returns an "Identity lacks required permissions" error in the `verify-cloud-auth-result` artifact.

- **What the user sees:** When the verify workflow fails, the Radius side panel reads the `verify-cloud-auth-result` artifact and shows: "The following recommended permissions are missing on the identity: <list of permissions>. Are you sure you want to create this environment without the recommended permissions?" The user is presented with two buttons: **Re-verify Credentials** to run the verify workflow again after correcting permissions, and **Create Environment** to bypass the permissions check.
- **Recovery:** The user adds the missing permissions and re-verifies, or bypasses the check and creates the environment anyway (for example, when permissions are managed elsewhere).

### Exception 4.5: Cluster or cloud provider cannot be reached

If the cloud provider or the named Kubernetes cluster (AKS on Azure or EKS on AWS) cannot be reached, the verify workflow returns a "Kubernetes cluster cannot be reached" or "Cloud provider cannot be reached" error in the `verify-cloud-auth-result` artifact.

- **What the user sees:** The user is shown an error on the **Create Environment** form stating which component was not reachable, with the same two buttons: **Re-verify Credentials** or **Create Environment** to bypass verification.
- **Recovery:** The user re-verifies with **Re-verify Credentials** after restoring reachability, corrects the cluster name, or bypasses verification and creates the environment anyway.

## Part 5: Deploy the application to the cloud

When the user is deploying an application to AWS or Azure, they are shown an animated application graph with each cloud resource moving through queued, in progress, success, or failure. Exceptions 5.1 through 5.4 are the Repo Radius outcomes surfaced to the user on the Radius side panel.

### Exception 5.1: Deployment failed, cancelled, or timed out

A deployment is running against a configured environment but does not complete: a resource fails (exposing the exact Radius error output), or the underlying workflow run is cancelled or times out. Remaining work does not run.

- **What the user sees:** The resources that did not deploy turn red on the application graph. For a resource failure, the failed node shows the exact Radius error output; for a cancelled or timed-out run, the red nodes show the error message "Deployment cancelled" or "Deployment timed out."
- **Resulting state:** A partial deployment; the application graph shows which resources were provisioned and which (in red) were not.
- **Recovery:** The user reads the failed node's error, or notes which resources are red, corrects the cause (definition or recipe), and redeploys to converge (Part 7).

### Exception 5.2: Cloud authentication or authorization failed

The user redeploys to an environment that was confirmed earlier, but cloud authentication or authorization now fails before any resource is touched, indicating the trust or permissions drifted since setup.

- **What the user sees:** The deployment fails immediately reporting that cloud authentication or authorization failed.
- **Recovery:** The user re-verifies the environment (Part 4), then redeploys.

### Exception 5.3: State could not be loaded

A redeploy that depends on prior state begins, but the persisted state cannot be read or restored, so the deployment cannot proceed. This does not apply to a first deployment, where no prior state exists.

- **What the user sees:** The deployment reports the persisted state could not be read or restored.
- **Recovery:** The user retries; if it persists, this is a Repo Radius state problem and the deployment cannot proceed until the state loads.

### Exception 5.4: State could not be saved

The deployment ran to completion and provisioned resources, but afterward the recorded state could not be written back, so the cloud may hold resources the recorded state does not reflect.

- **What the user sees:** The Radius side panel warns that the deployment ran but its recorded state could not be saved, and that orphaned resources may exist.
- **Resulting state:** Cloud resources exist but the recorded state is stale, so orphaned resources may exist.
- **Recovery:** The user redeploys to reconcile; where the recorded state cannot be brought back in sync, delete and redeploy (Part 8, then Part 5).

## Part 6: Modify the app to add a cache and open a PR

This part is a code-editing and pull request flow conducted entirely through the Copilot chat, not a Radius deployment. The user asks Copilot in natural language to make a change and open a pull request; Copilot works in the worktree and reports back in chat. Failures fall into three groups: Copilot cannot make the change, the pull request cannot be opened, or the post-merge view does not reflect the change.

### Exception 6.1: Code or definition change incomplete

The user asks Copilot to make a change, for example adding a cache, and Copilot edits the code and definition in the worktree but cannot complete it. No pull request is opened on an incomplete change.

- **What the user sees:** Copilot responds in chat: "I started adding the cache but couldn't finish the change, so I haven't opened a pull request yet. Want me to try again, or adjust what you're after?"
- **Recovery:** The user asks Copilot to try again, or refines the request (for example, names the service that should read from the cache); Copilot starts over in the worktree, so no partial change is left behind.

### Exception 6.2: Cannot open the pull request

Copilot has made the change in the worktree and the user asks it to open a pull request, but pushing the change or opening the pull request is rejected (a rejected push or insufficient permission). No pull request is opened.

- **What the user sees:** Copilot responds in chat: "I made the change, but I couldn't open the pull request: <push or PR error>. That usually means I don't have permission to push to <repository>. Give me write access, or point me at a fork you can write to, and I'll open it."
- **Recovery:** The user grants write access, or adds a fork they can write to, then asks Copilot to open the pull request again.

### Exception 6.3: Application graph diff cannot be computed

A pull request is open and the user asks Copilot how the application graph changed, but it cannot compute the diff between the main branch and the pull request. The code changes already in the pull request are unaffected.

- **What the user sees:** Copilot responds in chat: "I couldn't work out how the application graph changed between the main branch and this pull request: <reason>. Your code changes in the pull request are safe. If the main branch has moved, ask me to refresh and I'll try again."
- **Recovery:** The user confirms the application definition is valid on both the main branch and the pull request, then asks Copilot to compare again; if the main branch moved, they ask Copilot to refresh first.

### Exception 6.4: Merge conflict or merge fails

A pull request is open and the user asks Copilot to merge it, but it cannot be merged because of a conflict or a failing check. The pull request stays open.

- **What the user sees:** Copilot responds in chat: "I can't merge this pull request yet: <conflict or failing check>. Once that's resolved I'll merge it, or you can merge it on GitHub."
- **Recovery:** The user resolves the conflict or the failing check, then asks Copilot to merge, or merges on GitHub.

### Exception 6.5: Stale application graph after merge

The pull request has just merged, and the user asks Copilot to show the updated application graph, but it still shows the pre-merge graph.

- **What the user sees:** Copilot responds in chat: "That merged, but the graph I'm showing is still the pre-merge version. Give it a moment and ask me to refresh, and I'll read the latest from the main branch."
- **Recovery:** The user asks Copilot to refresh after the merge completes; the modeled application graph view reads from the merged definition on the main branch.

A graph diff that fails to render is presentation over the application definition, so it is recoverable by asking Copilot to refresh and does not affect the code changes already in the pull request.

## Part 7: Re-deploy the modified application to the cloud

Re-deploying a modified application has the same exception scenarios as Part 5 with the addition of one.

### Exception 7.1: Removed resource not deleted

The user removes a resource from the definition and redeploys, but the removed resource is not torn down automatically, because Radius deployment is incremental and only adds or updates resources rather than deleting removed ones.

- **What the user sees:** The Radius side panel identifies resources removed from the definition that still exist in the environment and prompts the user to delete each one explicitly.
- **Resulting state:** The removed resource still exists in the environment after redeploy.
- **Recovery:** The user clicks the removed but still-deployed resource on the application graph and selects delete, and that individual cloud resource is deleted.

## Part 8: Delete application deployments

Deleting an application has the same in-progress failure scenarios as Part 5 (Exceptions 5.1 through 5.4). Because deletion is destructive, it is gated by a confirmation dialog that summarizes the target application, environment, and resources to be deleted before teardown begins.
